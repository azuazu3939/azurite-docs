# Wiki執筆ガイド

> [!TIP]
> このガイドは、`wiki.md` を「短い共有メモ」ではなく「あとから育つ運用 wiki」にするための共通ひな形です。

## このガイドの目的

`summary.md` は実装の現状説明、`examples.md` は編集時の入口、`wiki.md` は人が読み足しやすい運用知識の集約先として使います。  
そのため `wiki.md` には、説明文だけでなく「あとから追記したくなる項目」を最初から置いておくのが重要です。

## 最低限ほしい章立て

| 項目 | 何を書くか | 役割 |
| --- | --- | --- |
| クイックデータ | 主担当、主設定、影響先、変更難度 | ページ冒頭の見取り図 |
| 概要 | その要素がゲーム内で何を担うか | 初見の導入 |
| プレイヤー体験 | プレイヤーがどの順で触るか | 体験設計の確認 |
| 構成の見取り図 | サブシステム、設定ファイル、runtime | 編集範囲の把握 |
| 編集フロー | どこから触るか、何を先に決めるか | 実作業の入口 |
| 連動する要素 | 変更が波及する相手 | 事故防止 |
| FAQ | よく迷う点、変更時の注意 | 運用知識の蓄積 |

## 書き足しやすい項目

以下の項目は、空欄でも見出しだけ置いておく価値があります。

| 項目 | 追記例 |
| --- | --- |
| 既知の癖 | 「旧ロア prefix を消し忘れると二重表示」 |
| 未解決メモ | 「Campaign 報酬倍率との責務分離を要整理」 |
| 監視ポイント | 「ログ、DB、Drive 同期、権限」 |
| 実装依存 | 「NMS 有効 role のみ runtime 起動」 |
| テスト観点 | 「プレイヤー導線、GUI、再ログイン、報酬受取」 |
| 将来の拡張候補 | 「ページ分割したい要素、詳細ページ化したい概念」 |

## おすすめの書き方

1. 冒頭に「これは何のページか」を 2 行で書く
2. その下にテーブルでクイックデータを置く
3. プレイヤー視点と運営視点を分ける
4. 最後に FAQ と関連リンクを置く

## 使える Markdown パターン

### 1. クイックデータ表

| 項目 | 内容 |
| --- | --- |
| 主設定 | `config.yml` |
| 主 runtime | `SomeRuntime` |
| 影響先 | 経済、釣り、鍛造 |

### 2. 注意書き

> [!WARNING]
> access-tag を変えると、resourceworld や forge、釣り戦闘側の利用条件にも波及します。

### 3. 手順

1. まずどのファイル層を変えるか決める
2. 次に連動要素を洗う
3. 最後にゲーム内導線で破綻がないか確認する

## 育て方の方針

- 1 ページで収まらなくなったら、まず `wiki.md` に概要だけ残して別ページへ分割する
- 変更履歴を時系列で積みすぎず、現行仕様に必要な知識を優先する
- 実装メモと運用メモを混ぜすぎない
